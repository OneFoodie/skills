# 专家团协作工作流
## 概述
功能定位
本技能是专家团协作的统一入口，根据任务复杂度智能调配专家资源，采用串行执行模式确保质量和可控性。适用于复杂业务场景、多阶段任务、需要多专家协作的交付场景。

核心原则
复杂度驱动：根据任务复杂度自动调配专家数量和执行模式

串行优先：默认串行执行确保质量，仅在完全独立场景下考虑并行

专家路由：按任务类型自动匹配最合适的领域专家

闭环交付：六步执行闭环确保从分析到复盘全程可控

适用场景
| 场景类型 | 说明 | 示例 |

|----------|------|------|

| 复杂业务需求 | 涉及多个业务模块、需要多轮交互 | 新增完整功能模块 |

| 跨领域任务 | 需要前后端协作、设计与实现配合 | 用户管理系统重构 |

| 高风险变更 | 涉及核心逻辑、需要严格验证 | 认证授权机制升级 |

| 多阶段交付 | 需要分阶段实施、逐步验证 | 数据库迁移 + 应用改造 |

禁用边界
简单任务：单一文件修改、配置调整（直接执行，无需专家团）

紧急修复：需要快速响应的问题（简化流程，事后复盘）

纯执行任务：无设计决策的机械性任务（直接调用对应专家）

任务复杂度分级
分级标准
| 等级 | 复杂度 | 判断标准 | 专家配置 | 执行模式 |

|------|--------|----------|----------|----------|

| L1 | 低复杂度 | 单一模块、<3 个文件、无外部依赖 | 1 个主导专家 | 直接执行 |

| L2 | 中复杂度 | 2-3 个模块、3-10 个文件、少量外部依赖 | 1 个主导 +1 个辅助 | 串行执行 |

| L3 | 高复杂度 | 3-5 个模块、10-20 个文件、多个外部依赖 | 2-3 个专家协作 | 分阶段串行 |

| L4 | 极高复杂度 | >5 个模块、>20 个文件、核心系统变更 | 专家团（3-5 个） | 多阶段串行 + 用户确认 |

复杂度评估维度
| 维度 | 评估项 | 分值 |

|------|--------|------|

| 模块数量 | 涉及的业务模块数 | 1 个=1 分，2-3 个=2 分，>3 个=3 分 |

| 文件变更 | 预计变更文件数 | <3=1 分，3-10=2 分，10-20=3 分，>20=4 分 |

| 外部依赖 | 依赖其他系统/服务数 | 无=0 分，1-2 个=1 分，>2 个=2 分 |

| 风险等级 | 对核心功能的影响 | 无影响=0 分，边缘功能=1 分，核心功能=3 分 |

| 验证成本 | 测试验证工作量 | <1h=0 分，1-4h=1 分，>4h=2 分 |

复杂度判定公式

总分 = 模块数量分 + 文件变更分 + 外部依赖分 + 风险等级分 + 验证成本分

L1（低复杂度）：1-3 分

L2（中复杂度）：4-6 分

L3（高复杂度）：7-9 分

L4（极高复杂度）：10-14 分

快速判定表（实践指南）
| 任务特征 | 快速判定 | 说明 |

|----------|----------|------|

| 修改单个组件/函数 | L1 | 影响范围局限在单文件 |

| 新增 API 接口 + 测试 | L2 | 涉及实现和验证两个环节 |

| 前端页面 + 后端 API | L2-L3 | 跨前后端，需评估页面复杂度 |

| 数据库表结构变更 | L3 | 涉及数据迁移和兼容性 |

| 认证/授权机制改造 | L3-L4 | 核心安全功能，影响面广 |

| 完整功能模块（增删改查 + 列表 + 详情） | L4 | 多文件、多模块、完整业务闭环 |

专家调配策略
L1 任务（低复杂度）
调配策略：单专家直接执行


用户指令 → 复杂度评估 (L1) → 路由到主导专家 → 执行 → 验证 → 交付

示例场景：

修复单个组件的样式问题

修改 API 接口的返回字段

更新配置文件

执行要点：

直接调用对应领域专家

简化流程，省略详细方案设计

完成后快速自检

L2 任务（中复杂度）
调配策略：1 个主导专家 + 1 个辅助专家，串行执行


用户指令 → 复杂度评估 (L2) → 制定方案 → 主导专家执行 → 辅助专家验证 → 交付

示例场景：

新增用户管理 API 接口

实现前端数据列表页面

数据库表新增字段

执行要点：

主导专家负责核心实现

辅助专家提供审核意见或完成配套工作

需要简单的方案设计

L3 任务（高复杂度）
调配策略：2-3 个专家协作，分阶段串行执行


用户指令 → 复杂度评估 (L3) → 制定方案 → 阶段 1(专家 A) → 阶段 2(专家 B) → 阶段 3(专家 C 验证) → 交付

示例场景：

用户认证系统重构

数据库迁移 + 应用改造

完整的前后端功能模块

执行要点：

必须制定详细方案

分阶段执行，每阶段完成后验证

需要用户确认关键决策点

L4 任务（极高复杂度）
调配策略：专家团（3-5 个专家），多阶段串行 + 用户确认


用户指令 → 复杂度评估 (L4) → 方案评审 → 阶段 1 → 用户确认 → 阶段 2 → 用户确认 → 阶段 3 → 全面验证 → 交付

示例场景：

新增完整业务模块（如绩效评价系统）

架构升级（如单体拆分为微服务）

核心安全机制改造

执行要点：

必须经过方案评审

每个关键阶段后设置用户确认点

全面验证（功能、性能、安全、合规）

详细记录和复盘

用户交互机制
必须询问用户的关键决策点
| 决策点 | 说明 | 示例问题 |

|--------|------|----------|

| 任务启动前 | 确认任务目标、范围、优先级 | “请确认本次任务的主要目标是？” |

| 复杂度评估后 | L3/L4 任务需确认方案 | “以下是任务方案，请确认是否继续？” |

| 方案选择 | 存在多个可行方案 | “以下三种方案，您倾向于哪种？” |

| 阶段转换 | L3/L4 任务阶段完成后 | “阶段 1 已完成，请确认是否进入阶段 2？” |

| 风险决策 | 涉及重大变更或潜在风险 | “此变更存在以下风险，是否继续？” |

| 验收标准 | 需要用户明确验收口径 | “您希望的验收标准是？” |

执行流程（六步闭环）
步骤一：分析指令
输入：用户指令、上下文信息

输出：目标确认、范围界定、复杂度评估、可执行假设

关键动作：

理解任务目标和范围

识别不确定项和需要澄清的问题

进行复杂度评估（L1-L4）

给出可执行假设（针对信息不足项）

复杂度评估检查表：

 涉及多少个业务模块？

 预计变更多少个文件？

 有多少外部依赖？

 是否影响核心功能？

 验证工作量多大？

用户交互点：

任务目标不明确时，使用 ask_user_question 澄清

L3/L4 任务需确认任务范围和优先级

步骤二：制定方案
输入：分析结论、复杂度评估结果

输出：任务拆分、专家调配方案、执行计划、验收标准

关键动作：

根据复杂度选择专家配置

拆分任务为可执行的子任务

定义执行顺序和依赖关系

设定验收标准和回滚点

异常处理与回滚
异常分类与处置
| 异常类型 | 场景 | 处置策略 |

|----------|------|----------|

| 信息不足 | 缺少必要信息 | 建立最小可执行假设并标注边界 |

| 需求冲突 | 多需求相互矛盾 | 记录冲突，执行最小影响兼容路径 |

| 验证失败 | 检查点未通过 | 进入修复分支并复检 |

| 专家执行失败 | 子代理执行超时/失败 | 重试 1 次 → 降级处理 → 回滚 |

回滚触发条件
满足以下任一条件时触发回滚：

| 条件 | 判定标准 |

|------|----------|

| 核心验收项未通过 | 必检项中有 2 项以上未通过 |

| 连锁失败 | 修复后新问题数 > 原问题数 |

| 关键路径阻塞 | 主流程阻塞超过 30 分钟 |

| 重复修复失败 | 同一问题修复尝试超过 3 次 |

| 资源污染 | 关键资源状态不可恢复 |

回滚策略
| 阶段 | 策略 |

|------|------|

| 可重试 | 单次失败可重试一次 |

| 降级处理 | 重试失败后简化目标继续 |

| 回滚中止 | 降级后仍失败则回滚并中止 |
